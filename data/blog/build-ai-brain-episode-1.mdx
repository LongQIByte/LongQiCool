## 先回答一个问题：为什么笔记越记越乱

如果你用 Obsidian 记了几百篇笔记，但真要找一个东西的时候，还是靠搜索关键词碰运气——这很正常，因为问题不在软件，而在方法。

笔记收藏进来的那一刻，它对你是有意义的。但如果没有被**激活**，它就死了——你不会再见到它，也不会跟它产生新的联系。

这里有个很简单的脑科学原理：**记忆不是靠存放，而是靠联系**。一个知识点被你提取、关联的次数越多，它在你脑中扎得就越深。任何学习本质上都是从已知推导未知，用你已经掌握的东西去反推新遇到的内容，这才是最高效的学习路径。

所以，我们要做的不是"整理"，而是让笔记从"死的档案"变成"活的系统"。

## 核心架构：raw + wiki 双层结构

我现在的知识库分两层，这个方法来自 AI 大牛 Andrej Karpathy 提出的 **LLM Wiki** 理念。

| 层级      | 职责                                                                                | 状态                         |
| --------- | ----------------------------------------------------------------------------------- | ---------------------------- |
| **raw/**  | 原始素材的仓库。PDF、网页剪藏、会议记录、随手拍的图片——原封不动丢进去。             | "死"的、不可变的素材层       |
| **wiki/** | AI 辅助消化的知识层。每篇文章围绕一个概念，有 frontmatter、双向链接、Sources 出处。 | "活"的，既给 AI 看，也给人看 |

**关键洞察**：原始的和提炼的必须分开。混在一起，噪音会大到没法用。

## 第一步：在 Obsidian 里搭好骨架

打开你的 Obsidian Vault，在根目录新建以下文件夹和文件：

```
.
├── raw/                 ← 原始素材，只增不改
├── wiki/                ← 消化后的知识文章
│   ├── index.md         ← 全库文集索引
│   └── log.md           ← Append-only 操作日志
├── projects/            ← 正在做的事（可选）
├── assets/              ← 配图资源（可选）
└── CLAUDE.md            ← 给 AI 看的"使用说明书"
```

### wiki 文章的标准格式

每篇 wiki 文章建议用以下 frontmatter：

```markdown
---
type: wiki
topic: 知识管理与认知工具
status: 活跃
source: [[raw/某PDF.pdf]]
created: 2026-04-16
updated: 2026-04-16
---

# 文章标题

正文内容...

## Sources

- [[raw/某PDF.pdf]]
```

### wiki/index.md 示例

```markdown
# Wiki 索引

## 知识管理与认知工具

- [[wiki/Obsidian与ClaudeCode知识库实践]] — 我的 AI 辅助知识库搭建实录
- [[wiki/第二大脑]] — 从笔记系统到外置大脑的概念梳理

## 编程与技术

- [[wiki/前端溢出解决思路]] — flex 布局下 Select 溢出的处理方案
```

### wiki/log.md 示例

```markdown
# 操作日志

[2026-04-16] init | 知识库 | 初始化 raw + wiki 双层结构
[2026-04-16] ingest | 第二大脑 | 读完《Building a Second Brain》，整理核心概念
```

## 第二步：写一份 CLAUDE.md 说明书

在 Vault 根目录创建 `CLAUDE.md`，这是整个系统的"宪法"。Claude Code 每次进入项目时都会先读它。

下面是一份可以直接抄的最小模板：

````markdown
# Workspace — Personal LLM Wiki

## 目录职责

- `raw/` — 原始素材，**不可变**。Claude 只读，绝不修改。
- `wiki/` — Claude 维护的消化笔记。每篇文章 `.md` 格式，围绕概念组织。
- `projects/` — 正在做的事。
- `assets/` — 配图资源。

## 搜索优先级

当用户提问或要求检索信息时，**优先搜索本资料库内的对应目录**：

- `projects/` — 项目、可执行任务相关
- `wiki/` — 知识、已消化的笔记
- `raw/` — 原始素材、参考资料

## 三个触发行为

### 触发 1: Ingest（录入素材）

**触发词**："收到 xxx"、"ingest 这个"。

运作：
a. 把素材存进 `raw/` 目录对应主题子目录。
b. 更新 `wiki/<文章名>.md`。
c. 检查冲突：新素材和已有内容矛盾时，在文章底部加 `[:!conflict:]` 批注。
d. 更新 `wiki/index.md`。
e. 更新 `wiki/log.md`。

### 触发 2: Update（更新已有内容）

**触发词**："更新 xxx"、"新进展"。

运作：
a. 定位目标文件。
b. 增量写入：在最相关小节追加新认知；如属纠错，用删除线保留旧文并标注原因。
c. 检查冲突。
d. 更新 `wiki/index.md`。
e. 更新 `wiki/log.md`：

```
[YYYY-MM-DD] update | <主题> | <简短说明>
```

### 触发 3: Archive（归档旧内容）

**触发词**："归档 xxx"、"过时了"。

运作：
a. 在原文顶部加 `status: archived` frontmatter。
b. 更新 `wiki/index.md`，标注 `(Archived)`。
c. 更新 `wiki/log.md`：

```
[YYYY-MM-DD] archive | <主题> | <归档原因>
```
````

你可以根据实际需求继续扩展这份说明书。原则是：**越清晰，AI 的执行就越稳定**。

## 第三步：让 Claude Code 参与日常

我使用的是 [Claude Code](https://claude.ai/code)，它可以直接读写我电脑上的本地文件。下面是三个最常用的场景。

### 场景 1：录入新素材

今天读到一篇关于 Agent 架构的新文章，我的动作只有两步：

1. 把文章（或 PDF）丢进 `raw/` 的对应子目录；
2. 在 Claude Code 里说：
   > "这是新素材，帮我把关于 Agent 架构的部分更新到 wiki 里。"

Claude 会：

- 先读 `CLAUDE.md` 了解规则；
- 扫描 `wiki/` 里是否已有相关文章；
- 判断是新建概念页，还是补充到已有页面；
- 最后更新 `wiki/index.md` 和 `wiki/log.md`。

### 场景 2：提问与检索

不是去网上搜标准答案，而是直接问自己的知识库：

> "我之前关于强化学习的训练心得都写在哪儿了？"

Claude 会直接读取你本地的 markdown 文件，基于你自己的笔记来回答，还会把关联笔记的链接一起指出来。这个过程本身就是一次"提取"，旧知识被重新激活了。

### 场景 3：维护索引

最省心的一点是：每次更新完，Claude 会顺带检查 `wiki/index.md` 和 `wiki/log.md` 有没有漏掉。整个系统的结构不会随时间变乱，这是人工维护很难做到的。

## 起步阶段的最小行动建议

你不需要一次性把几百篇旧笔记都翻新。我的建议是：

1. **先在 Obsidian 里分出 `raw/` 和 `wiki/` 两个文件夹**。
2. **写一份简单的 `CLAUDE.md`**，哪怕只有上面模板的一半长。
3. **挑一篇你最近读过的文章或一本书的笔记**，丢进 `raw/`，然后让 Claude 帮你改成概念化、带链接的 wiki 格式。
4. **坚持 append-only 的 `log.md`**，每次操作后记一笔，未来回看时你会清楚知道这个系统是怎么长大的。
5. **试一次提问**：基于 wiki 里的内容，向 Claude 提一个你自己都差点忘了的问题，感受"知识被提取"的过程。
