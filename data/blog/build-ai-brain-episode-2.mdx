## 先搞清一件事：手机在你的知识库里是什么角色

对很多人来说，**手机才是最高频的录入设备**，也是轻量检索的第一入口。

如果知识库只能坐在电脑前维护，真实利用率往往会断崖式下跌——灵感出现在地铁上、想法发生在睡前、想确认某件事发生在走路时，这些时刻你离电脑都很远。

所以这期不讲更多概念，只回答一个问题：**离开电脑，怎么继续让「本地知识库」长大？**

我的答案很直接：**用手机说话、用手机追问**，把内容交给跑在电脑上的 Claude Code，让它按你在 `CLAUDE.md` 里约定好的规则，写进日记、心得或 wiki。路上和床边的交互，**走 Happy 这条线就够了**——第二期讲的是怎么把日常最高频的设备接到你的本地大脑上，而不是教大家用手机上的 Obsidian 当「读长文、逛图谱」的主力。

## 为什么我更在意「短链路」和「可解释」

市面上不少「AI 笔记」方案，链路是：App → 中转服务 → 向量库 / 大模型 → 再回写到某个黑盒存储。

这类方案不是不能做，但对**个人本地知识库**来说，我经常遇到三类摩擦：

- **链路一长，排错就痛苦**：任意一环抖动，你都要猜是网络、服务还是同步逻辑的问题。
- **维护不直观**：笔记最终落在哪、被改成了什么形态，不一定一眼能看清。
- **把太多能力塞进同一个手机 App**：录入、检索、精读、图谱全挤在一起，往往链路更长、更不可预期；这期宁可**手机上只保留 Happy 这一刀**，执行与归档交给本机 Agent，文件形态仍是你熟悉的 Markdown 仓库。

所以这期的取向是：**可控、直观、尽量少中转**。我不是在追求「全自动魔法」，而是在追求「我随时知道数据在哪、长什么样」。

## 这期的链路：Happy → Claude Code → 本地 Vault

我用的是 **Happy**：它作为手机上的语音 / 文字入口，把你说的话发到**本机正在运行的 Claude Code**。

对你来说，动作仍然是第一期那一套——Claude 读 `CLAUDE.md`，在 `raw/` 与 `wiki/` 之间按规则写入；区别只是**触发方式**从「坐在终端前打字」变成了「拿着手机说几句」。

只要电脑上的 Claude Code 在跑、Happy 的桌面端服务在跑，手机侧用蜂窝网络也可以发（不必和电脑同一个 Wi‑Fi）。数据默认仍然落在你的 Vault 里，**没有「手机一份、电脑一份」的同步焦虑**——大脑仍在你的机器上，手机是入口与遥控器。

## 场景一：通勤路上，读完文章后的口述心得

假设你在地铁上读到一篇讲 Transformer 的好文，手头没有纸笔。

我的习惯是：**不要把原文整篇丢进知识库就算完**，而是读完之后，用自己的话口述——这篇文章的关键点是什么、它让你联想到你正在做的哪件事、对你下周的工作有什么启发。

具体做法：

1. 打开手机上的 Happy，按住语音输入（输入法我用豆包，识别速度和准确率都不错）。
2. 直接说你的理解与联想。
3. 补一句指令，让 Claude 知道往哪写，例如：
   - 「请记到我今天的日记里，并整理进个人心得。」
   - 或者：「放进 wiki，按我 Vault 的规则建链接、更新索引。」

Claude 会在本地读现有笔记，判断是**新建概念页**还是**补全已有页面**，然后更新 `wiki/index.md`、`wiki/log.md`——这和你在电脑里对它说话，是同一套流程。

## 场景二：睡前，今日复盘与明日排期

第二个高频场景是睡前：躺在床上，把今天的工作顺一遍，把明天要先做什么、可以延后什么，口述出来。

例如：「明天上午先对齐需求文档，下午开发先不动，因为接口还没定……」

这个过程中，你已经在**帮自己梳理优先级**；同时，这些内容可以让 Claude 按你的规则写进日记或项目笔记，相当于一个**随身携带的助理**，帮你把「想过的事」落成「可检索的文字」。

## Happy 是什么，以及怎么装

先看官网说明：[happy.engineering](https://happy.engineering/)。

**电脑端**装起来就一行（全局安装并启动服务）：

```bash
npm i -g happy-coder && happy
```

**手机端**（与官网说明一致即可）：

- **iOS**：在苹果 **App Store** 搜索 Happy 并安装（以商店里的应用名为准）。
- **Android**：在 **Google Play** 商店下载安装。

装好后与正在运行上述服务的电脑完成配对（细节以 [happy.engineering](https://happy.engineering/) 当前文档为准）。

装好以后：**电脑保持 Claude Code 可用、Happy 桌面端在跑**，手机就能直接把语音或文字请求送到本机 Claude，由它读写你的 Vault。

## 起步阶段的最小行动建议

1. **若还没有第一期那套底座，先别急着装 Happy**：按 [第一期：双层目录 + CLAUDE.md + 日常协作流程](/blog/build-ai-brain-episode-1)从零跟一遍；若想少手工、让本机 Agent 直接帮你拉齐目录和配置，跟这篇 [《Obsidian + Claude Code 知识库自动初始化脚本》](/blog/obsidian-ai-knowledge-base-init) 做即可（**Claude Code** 或 **Cloud Code** 都能照里面的步骤执行）。确认 `CLAUDE.md`、`raw/` / `wiki/` 已在，且 Ingest / Update 在**电脑上**跑通。
2. **装好 Happy 双端**，用手机发一句最简单的测试：「帮我在 log 里记一条：移动端链路已打通。」
3. **试一次「口述 → wiki」**：故意说一个会触发新建概念页的主题，回头在电脑上打开 Vault，看文件与索引是否按规则落好。
4. **试一次「口述 → 日记」**：把第二天的三件待办说进去，第二天在电脑上打开日记是否一目了然。

## 写在最后

第二期想传达的，其实很朴素：**大家每天摸得最多的是手机，本地知识库也应该能用手机「用起来」。**  
这一环不是靠「再装一个笔记 App 当阅读器」，而是靠 **Happy**：说话录入、随口让本机 Claude Code 去搜、去写、去改——大脑仍在你的电脑上，手机只是你最顺手的遥控器和话筒。

第三期我会聊：**怎么让这个系统随着笔记变多，慢慢长出更贴近你的「个人画像」，并辅助决策**——如果你愿意跟这个系列一起往前走，先把 Happy 这一环跑通，后面的迭代会顺很多。
